## When to use

Use Combobox when scanning the full list would be slow and a text query or quick
keyboard search materially improves selection.

Avoid Combobox when the list is tiny, non-searchable, or when a standard Select
already keeps the interaction simpler.

Use the multiple variant only when the selected summary stays understandable in

## Composition patterns

Keep the search field, results list, and empty state together as one compact
selection surface. Use surfaced composition when the control belongs to a
contained panel.

## Accessibility contract

- the trigger exposes combobox semantics and reflects expanded state
- the open layer keeps the search input and results list in the same focus flow
- options remain exposed with `option` semantics and visible labels
- keyboard users can open, search, navigate, and commit a selection
- multiple selection still closes predictably after each committed choice

## Validation

What is checked automatically:

- Storybook runs an accessibility scan scoped to the Combobox example area, including the open portal content
- Storybook interaction checks verify portal mounting, open-state semantics, visible search input, selection, and close behavior
- UI tests cover single and multiple selection flows, controlled state, filtering, disabled items, slot hooks, refs, class contracts, and a11y

What is checked manually:

- keyboard smoke pass for search, arrow navigation, and selection
- visual pass for trigger summary text, cursor behavior, and long-result wrapping

## Responsive notes

The trigger summary and search results must stay readable at narrow widths. Long
labels should wrap instead of clipping, and the open list should stay scrollable
without introducing horizontal overflow.
