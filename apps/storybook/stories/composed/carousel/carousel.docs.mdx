## When to use

Use Carousel for tightly curated sequences such as meal plans, compact reports,
or card stacks that benefit from previous and next controls.

Prefer ScrollArea when users need continuous scrolling instead of snap-based
navigation.

## Accessibility contract

- the root renders as a labelled `region` with `aria-roledescription="carousel"`
- the carousel root is keyboard reachable and handles arrow keys in the current
  orientation
- previous and next controls expose stable button labels
- slides keep a stable `data-slot` marker plus `aria-roledescription="slide"`
- default `data-slot` hooks remain stable on the root, viewport, content,
  slides, and controls

## Validation

What is checked automatically:

- Storybook runs an accessibility scan scoped to the local carousel audit area
- Storybook interaction checks verify stable slot hooks, keyboard focus, arrow
  navigation, and next/previous updates against the active-snap readout
- UI tests cover region semantics, button actions, focusable keyboard target,
  vertical layout classes, `setApi`, stable slot hooks, refs, and a11y

What is checked manually:

- visual pass for card clipping, control visibility, and spacing at 320, 390,
  768, and 1024 widths
- keyboard smoke pass to confirm the focused root and button controls remain
  usable in both orientations

## Responsive notes

The default controls sit outside the viewport, so compact layouts should leave
inline or block padding around the carousel root. The responsive stress story
keeps that space explicit so controls stay visible at mobile widths.
