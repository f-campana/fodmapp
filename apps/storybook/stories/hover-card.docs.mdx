## When to use

Use HoverCard for compact previews, richer food summaries, or context blocks
that benefit from a title, body copy, and secondary detail.

Avoid HoverCard for mandatory tasks, full editing flows, or content that must
remain open while users navigate elsewhere.

## Accessibility contract

- keyboard focus can open the hover card without relying on hover alone
- moving focus away closes the open surface cleanly
- portal content can mount into a local container for isolated checks
- the arrow, content, and trigger keep their documented slots in default composition

## Slot hooks

- in default composition, `data-slot` hooks stay stable on the root, trigger, portal, content, and arrow
- when using `HoverCardTrigger` with `asChild`, child attributes take precedence, including `data-slot`

## Validation

What is checked automatically:

- Storybook runs an accessibility scan scoped to the local hover-card audit area
- Storybook interaction checks verify focus open, local container mounting, the `asChild` slot exception, arrow rendering, and blur close
- UI tests cover default-composition slot stability, the `asChild` slot exception, local container mounting, semantic classes, ref forwarding, and a11y

What is checked manually:

- visual pass for trigger-to-panel spacing and non-touching borders
- narrow-width review to confirm the open card remains visible and readable

## Responsive notes

Hover card content keeps a capped width and collision padding so longer
descriptions can wrap without clipping the panel on smaller screens.
